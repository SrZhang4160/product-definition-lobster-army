🚦 检查点 B：语义一致性校验

龙虾三是串行链条的源头。如果它偏离了用户 Idea，后续龙虾全部失效。这个检查点用 Haiku 做轻量级校验，成本 ~$0.005。

// gate_check.js — 龙虾三输出后执行 async function gateCheck(anchor, lobster3Output) { const prompt = ` 原始产品 Idea：${anchor.product_anchor} 龙虾三产品定位句：${lobster3Output.one_liner} 这两者语义是否一致？打 0-10 分。 只输出 JSON：{"score": N, "reason": "..."} `; const result = await callHaiku(prompt, { max_tokens: 100 }); if (result.score < 5) { // 第一次低于阈值：换 temperature 0.5 重跑龙虾三 const retry = await rerunLobster3({ temperature: 0.5 }); const retryCheck = await gateCheck(anchor, retry); if (retryCheck.score < 5) { // 连续两次失败 → 中止，返回部分报告 + 告警 return { status: "aborted", partial: true }; } } return { status: "passed", score: result.score }; }

Orchestrator 核心代码结构（v3 硬化版）

// orchestrator.js v3 — 状态机 + 容错 + 断点续跑 const STATES = { INIT: "init", ANCHOR: "anchor", PARALLEL: "parallel", COMPRESS: "compress", LOBSTER_3: "lobster_3", GATE_CHECK: "gate_check", LOBSTER_4: "lobster_4", LOBSTER_5: "lobster_5", ASSEMBLE: "assemble", DONE: "done", FAILED: "failed", PARTIAL: "partial" }; async function runAnalysis(idea, resumeFrom = null) { const run = resumeFrom ?? createNewRun(idea); const config = loadConfig(); try { // ── Phase 0: 锚点生成（Haiku LLM，非模板）── if (run.state <= STATES.ANCHOR) { run.anchor = await withRetry(() => generateAnchorLLM(idea, config), { maxRetries: 2 }); validateSchema(run.anchor, anchorSchema); // 硬校验 persistState(run, STATES.PARALLEL); // 状态持久化 → 断点续跑 } // ── Phase 1: 并行龙虾一 + 二 ── if (run.state <= STATES.PARALLEL) { const [out1, out2] = await Promise.allSettled([ withRetry(() => runLobster("L1", { anchor: run.anchor, config })), withRetry(() => runLobster("L2", { anchor: run.anchor, config })), ]); run.L1 = handleSettled(out1, "L1"); // 一只失败不阻塞另一只 run.L2 = handleSettled(out2, "L2"); persistState(run, STATES.COMPRESS); } // ── Phase 2: 独立压缩 + Schema 校验 + Fallback ── if (run.state <= STATES.COMPRESS) { const [s1, s2] = await Promise.all([ // 各自独立压缩（并行） compressWithFallback(run.L1, config), compressWithFallback(run.L2, config), ]); run.summary = mergeAndValidate(s1, s2, run.anchor); persistState(run, STATES.LOBSTER_3); } // ── Phase 3: 龙虾三 + 熔断检查点 ── if (run.state <= STATES.LOBSTER_3) { run.L3 = await withRetry(() => runLobster("L3", { anchor: run.anchor, summary: run.summary, config })); persistState(run, STATES.GATE_CHECK); } if (run.state <= STATES.GATE_CHECK) { const gate = await gateCheck(run.anchor, run.L3); if (gate.status === "aborted") { return persistState(run, STATES.PARTIAL); // 中止，返回部分结果 } run.gateScore = gate.score; persistState(run, STATES.LOBSTER_4); } // ── Phase 4: 龙虾四（注入龙虾三全文 + 市场摘要）── if (run.state <= STATES.LOBSTER_4) { run.L4 = await withRetry(() => runLobster("L4", { anchor: run.anchor, context: run.L3, // 龙虾三全文 marketSummary: run.summary, // 龙虾一二市场摘要 config })); persistState(run, STATES.LOBSTER_5); } // ── Phase 5: 龙虾五（深度批判，读全文）── if (run.state <= STATES.LOBSTER_5) { run.L5 = await withRetry(() => runLobster("L5", { anchor: run.anchor, fullL3: run.L3, // 龙虾三全文（推理链条） fullL4: run.L4, // 龙虾四全文（方案细节） summaries: run.summary, // 龙虾一二摘要 config })); persistState(run, STATES.ASSEMBLE); } // ── Phase 6: 拼接报告 + 质量元数据 ── const report = assembleReport(run); report.meta = buildMeta(run); // token/成本/耗时/各检查点分数 report.confidence = calcConfidence(run); // 各龙虾置信度标注 saveAll(run, report); persistState(run, STATES.DONE); } catch (err) { run.error = err.message; persistState(run, STATES.FAILED); emitAlert(run); // 通知监控系统 } }

02内存管理架构

龙虾之间不能直接对话。所有信息传递都通过三层内存体系完成，确保每只龙虾在有限上下文窗口内获得最关键的上游信息。

🔥 热内存
~6,000 tokens
内容：当前龙虾的 System Prompt + Few-shot 示例 + CoT 指令
特点：每只龙虾独立加载，互不干扰
锚点注入：每只龙虾的热内存开头都有相同的 product_anchor（~100 tokens），强制聚焦
管理方式：静态模板，从 /prompts/vN/ 目录加载

防漂移四道防线

🎯
防线一：锚点层
每只龙虾 Prompt 开头强制注入

统一格式的产品描述（name / target_user / core_problem），~100 tokens，由 Orchestrator 生成一次，注入所有龙虾。

🔁
防线二：摘要回声
每份摘要重复 product_anchor

压缩摘要 JSON 的第一个字段永远是 product_anchor，下游龙虾读到的每份输入都在反复强化上下文。

✅
防线三：输出自检
每只龙虾输出末尾的防漂移检查

要求龙虾在输出最后回答："你的分析是否100%围绕 {product_anchor}？" 偏离时自动标记。开发阶段使用，生产阶段可移除。

🧹
防线四：上下文节食
信息越少越聚焦

实际使用不到 Sonnet 4.6 窗口的 6%。不因为窗口大就往里塞更多东西——信息越少，龙虾越不容易跑偏。

关键设计原则

即写即存

每只龙虾完成后立即写 JSON 到 /runs/，不等全链路结束。中途崩溃不丢已完成的数据。

Schema 强约束

压缩摘要用 JSON Schema 验证。如果 Haiku 返回的 JSON 不合格，自动重试一次。

完整可回溯

meta.json 记录每次运行的完整配置，任何一次历史分析都可以精确复现。

全链路联调流程

阶段目标参与人验收标准

E2E 最终报告整体质量 PM 做整体评审，PE 监控 token/成本 5 个 Idea 端到端平均分 ≥ 6.5/10

阶段二 · 单龙虾调通（第 2-4 周）
从龙虾一开始，逐只攻克

里程碑：全链路能跑通一个 Idea，各龙虾输出之间有明确的上下游引用关系

PM：确认迁移后输出质量不退化（A/B 测试）
里程碑：CrewAI 版本质量 ≥ 手动版本，开发效率提升
阶段六 · 自迭代闭环（第 11-16 周）
评估龙虾 + DSPy 优化

里程碑：系统能自动评估 + 半自动优化 Prompt

Few-shot 迭代方法

📚
PM 负责：收集源材料

从麦肯锡 / CB Insights / YC 等来源找到真实的好报告。不复制内容，而是抽取结构框架：这份报告包含哪些模块、数据如何组织、结论如何呈现。每只龙虾需要 5-8 份源材料作为候选池。

🔬
PE 负责：抽象 + A/B 测试

将源材料抽象为结构化的 Few-shot 示例（~1,000 tokens/个）。初始选 3 个，跑 5 次看效果。然后换掉效果最差的样本，替换为新的，再跑 5 次对比。每只龙虾最终保留 2-3 个经过验证的"黄金样本"。

⚠架构 Review：12 个优化点

对需求文档 v1.0 和系统蓝图 v2.0 的交叉审计。按严重程度分为 🔴 架构缺陷（不改会出事）、🟡 设计盲区（影响质量上限）、🟢 成本/效率优化（锦上添花）。

🔴 架构缺陷（4 个）— 必须在阶段一解决

1
压缩摘要层是单点故障
Haiku 返回非法 JSON → 龙虾三四五全部瘫痪
问题当前设计只说"Haiku 压缩为 JSON"，但没有定义：Haiku 返回格式错误怎么办？字段缺失怎么办？product_anchor 被篡改怎么办？
影响压缩层失败 → 龙虾三收到损坏的上下文 → 整条串行链全部输出垃圾。一个 $0.01 的步骤拖垮整个 $0.50 的流程。
修复方案 三层防护：
① JSON Schema 硬校验：Orchestrator 用 JSON Schema（ajv / jsonschema）验证 Haiku 输出。缺字段 → 自动重试一次。
② Fallback 降级：重试仍失败 → Orchestrator 用规则模板从龙虾一/二原始输出中提取关键字段（正则匹配 TAM/SAM/SOM 数字），生成"最小可用摘要"。
③ 锚点不可变：product_anchor 由 Orchestrator 注入，不经过压缩层。压缩 Prompt 明确指令"不要修改 product_anchor 字段"。
2
龙虾五只读摘要，批判深度不够
摘要丢失了推理过程，"魔鬼代言人"找不到逻辑链上的真正漏洞
问题需求文档定义龙虾五读取"全部前四只龙虾的摘要"（~2,000 tokens）。但龙虾五的核心任务是找逻辑漏洞和致命假设——这需要看到推理过程，不只是结论。摘要 "TAM $120B" 看不出推导是否合理。
影响龙虾五的"魔鬼代言人"和"关键假设清单"模块质量上限被卡死。只能做表面批判，无法戳穿推理链条中的薄弱环节。
修复方案 给龙虾五开"全文阅读权限"，但分层注入：
① 摘要层（~2,000 tokens）：全部龙虾的结构化结论 → 放在 System Prompt 末尾
② 龙虾三全文（~3,500 tokens）：产品定义完整逻辑 → 放在 user message 中
③ 龙虾四全文（~5,000 tokens）：技术方案完整细节 → 放在 user message 中
总 input ~14,500 tokens，仍远在 Sonnet 4.6 窗口预算内。Output 建议从 3,000 提到 4,000 tokens。
成本增加：~$0.04/次，但风险分析质量有质的提升。
3
龙虾四缺少上游市场数据
技术方案的成本估算没有市场规模锚定，容易"空中楼阁"
问题当前设计：龙虾四只读龙虾三全文。但龙虾四要输出"BOM 成本 / 月运营 / 所需融资"——这些数字需要市场规模（来自龙虾一）和竞品定价（来自龙虾二）作为参照系。
影响龙虾四可能估出"种子轮需要 $2M"却不知道 TAM 只有 $10M，导致投入产出比明显不合理但无人发现（直到龙虾五来批判，但龙虾五也只有摘要）。
修复方案 给龙虾四注入 combined_summary.json（~500 tokens），让它同时看到龙虾三全文 + 龙虾一二的市场/竞品摘要。
总 input 从 ~8,000 变为 ~8,500 tokens，成本几乎不变。
在 Prompt 中明确指令："你的成本估算必须参照市场规模（TAM/SAM）和竞品定价，确保投入产出比合理。"
4
缺少级联熔断机制
龙虾三输出质量差 → 龙虾四五跟着全崩，白烧 token
问题串行链条中，如果龙虾三的产品定义偏离了用户 Idea（比如把"医疗影像 AI"理解成了"医院管理系统"），龙虾四和五会基于错误的产品定义继续工作，浪费约 $0.25。
影响当前没有任何中间质量检查。只能等全链路跑完、人工看最终报告才发现问题，此时 token 已经花完了。
修复方案 在龙虾三输出后加一步"轻量级校验"：
用 Haiku 做一次快速检查（~$0.005）：比较龙虾三的产品定位句和原始 Idea 的语义相关度。
输出一个 0-10 分。分数 < 5 → Orchestrator 自动重跑龙虾三一次（换稍高 temperature 0.5）。
连续两次 < 5 → 中止运行，记录错误，返回部分结果 + 告警。
成本：每次 ~$0.005，相当于整个流程的 1%。换来的是避免 50% 的 token 浪费。

🟡 设计盲区（4 个）— 影响输出质量上限

5
web_search 结果无质量过滤
搜索返回的内容可能过时、不相关、或充满 SEO 垃圾
问题需求文档说龙虾一做 3 次搜索、龙虾二做 5 次。但没有定义：搜索返回的内容如何筛选？如果返回的是一篇 2019 年的过时文章？如果返回 5000 tokens 的无关内容？
修复方案 ① 搜索结果截断：每次搜索结果限制在 800 tokens 以内（Orchestrator 层处理），超出部分截断。
② 来源年份过滤：在搜索 query 中追加年份关键词（如 "2024 2025"），搜索结果中明确标注来源年份。
③ Prompt 层防御：在龙虾一/二的 Prompt 中加入"如果搜索结果无法验证某个数据点，明确标注为 [未验证] 而非编造来源"。
6
只有龙虾一有 Rubric，其他四只没有
无法系统性评估龙虾二到五的输出质量
问题需求文档只为龙虾一定义了 4 维 Rubric（数据具体性 / 中国市场深度 / 逻辑自洽 / 可操作性）。蓝图沿用了这套评分体系，但龙虾二到五的输出性质完全不同——竞品矩阵、产品定位、技术方案、风险分析各自需要不同的评价维度。
修复方案 PM 的核心交付物应该包含 5 套独立 Rubric：

龙虾二：竞品覆盖度(3) · 数据时效性(3) · 差异化洞察(2) · CAC 合理性(2)
龙虾三：定位句精准度(3) · 旅程图完整性(2) · 功能优先级合理性(3) · 与上游数据一致性(2)
龙虾四：方案可行性(3) · 成本估算合理性(3) · 三套方案梯度清晰(2) · 时间线现实性(2)
龙虾五：批判深度(3) · 假设识别准确性(3) · 融资建议具体性(2) · 风险场景现实性(2)

另外需要一套端到端一致性 Rubric：龙虾四的方案是否响应龙虾三？龙虾五的风险是否针对龙虾四？
7
锚点生成用模板可能不够
用户输入的 Idea 千差万别，模板难以统一提取结构化信息
问题蓝图设计中锚点由 Orchestrator "格式化模板"生成。但用户输入可能是"做一个让医生更快看片子的 AI"，也可能是"reinvent how radiologists interact with PACS systems"。模板很难稳定地从这两种输入中提取 name / target_user / core_problem。
修复方案 用 Haiku 做锚点生成（~$0.003/次）：
输入用户 Idea → Haiku 输出固定格式的锚点 JSON：
{"name": "...", "target_user": "...", "scenario": "...", "core_problem": "..."}
加 JSON Schema 验证。这一步成本极低，但能显著提升锚点质量，尤其对模糊或英文输入。
8
评估龙虾"自己评自己"的可信度问题
LLM 评估 LLM 输出有已知偏见：倾向给高分、对格式好的内容更宽容
问题需求文档阶段三引入评估龙虾做自动打分。但 LLM-as-Judge 有系统性偏见：偏好长输出、偏好格式整齐的输出、对"看起来专业"但内容空洞的文本打高分。尤其"数据可验证性"维度——评估龙虾无法真正验证一个数据来源是否真实。
修复方案 限制评估龙虾只评"可机器验证"的维度：
✅ 结构完整性：是否包含所有要求模块（可用正则检查）
✅ 格式规范：JSON 是否合法、数字是否附带来源标注
✅ 下游一致性：龙虾四提到的产品名是否与龙虾三一致
❌ 数据可验证性：评估龙虾无法真正验证数据，改为"是否标注了来源年份"
❌ 结论可操作性：这是主观判断，必须保留人工评估

底线：人工评估至少持续到积累 30 组标注数据之后，不要过早依赖自动评估。

🟢 成本与效率优化（4 个）— 锦上添花

9
Prompt Caching：最大的降本机会
每只龙虾有 4,000-6,000 tokens 的静态内容，每次都重复发送
现状System Prompt (~800) + Few-shot (~3,000) + CoT 指令 (~200) = ~4,000 tokens/龙虾，每次运行都全量发送。5 只龙虾 + 压缩层 = ~25,000 tokens 的重复 input。
优化方案 Anthropic Prompt Caching：将 System Prompt + Few-shot 部分设为 cacheable。
Cache hit 只收标准价的 10%（$0.30/M 而非 $3/M）。
5 分钟 TTL 缓存写入成本 1.25x，但第二次调用起就开始省钱。

预估节省：
· 每次运行 ~25,000 tokens 的静态内容 × $2.7/M 节省 = ~$0.068/次
· 每月 300 次 = 省 ~$20/月（约总成本的 13%）
· 开发调试阶段（频繁调用同一 Prompt）节省更明显
实现方式 在 API 调用中将 system message 标记为 cache_control: {"type": "ephemeral"}。
注意：可缓存内容必须 ≥ 1,024 tokens（Sonnet）或 ≥ 2,048 tokens（Haiku）。Few-shot 部分刚好满足。
10
龙虾一二的并行可以更激进
锚点生成 + 压缩摘要也可以并行化
现状当前设计：生成锚点 → 并行跑龙虾一二 → 压缩摘要。这三步是串行的。
优化方案 步骤一（并行）：锚点生成（Haiku）和 龙虾一二的 API 调用可以同时发出——因为锚点 ~100 tokens，Haiku 返回极快（<1s），实际上龙虾一二还在等搜索结果时锚点就已经生成好了。可以将锚点用 Promise.race 的方式在龙虾一二真正需要前注入。

步骤二（并行）：龙虾一完成后，不等龙虾二——立即开始压缩龙虾一的摘要。龙虾二完成后立即压缩龙虾二摘要。两份摘要独立压缩，最后 Orchestrator 合并为 combined_summary.json。

预估节省：端到端延迟从 ~45s 降至 ~35s（省 ~10s）。

Batch API 适合回归测试
批量跑 10 个测试 Idea 时，用 Batch API 省 50% 成本
场景每次 Prompt 迭代后要用全部 10 个测试 Idea 做回归测试。如果实时跑 = 10 × $0.50 = $5.00。
优化方案 回归测试不需要实时结果（可以等几个小时）。使用 Anthropic Batch API，所有请求半价。
10 个 Idea 的回归测试：$5.00 → $2.50。
如果每周做 2 次回归测试，每月省 ~$20。

限制：Batch API 的结果可能需要数小时，不适用于生产环境的实时分析。

缺少运行监控仪表盘
随着运行次数增加，需要可视化质量趋势和成本趋势
问题当前 meta.json 记录了每次运行的 token/成本/耗时，但没有设计如何消费这些数据。100 次运行后，你有 100 个 JSON 文件但无法快速回答"上周质量比前一周好了吗？"
优化方案 阶段二实现一个简单的监控页面：
· 从 /runs/*/meta.json 聚合数据
· 展示：每日成本趋势、每只龙虾平均评分趋势、Token 使用量分布、web_search 失败率
· 用 SQLite（你已经计划接入）存储聚合数据
· 前端用 Recharts 或 Chart.js 可视化

这个仪表盘是 PM 做质量决策的重要工具，也是向投资人展示"系统在持续优化"的证据。

09容错设计

生产级系统的核心要求：任何单点失败不能导致整个流程崩溃。每一步都有明确的失败处理路径。

Orchestrator 状态机

整个流程被建模为一个有限状态机。每完成一步，Orchestrator 把当前状态写入磁盘（persistState）。系统崩溃后可以从上次保存的状态断点续跑，不需要重头开始。

架构扩展核心原则

配置驱动

龙虾定义、工具注册、评估模块、执行拓扑全部通过 JSON 配置。改配置不改代码。Orchestrator 是通用引擎，不包含任何龙虾特有逻辑。

渐进增强

每个扩展都可以独立启用/禁用。新工具有 fallback，新龙虾有 enabled 开关，新评估模块默认 off。不会因为新功能导致现有流程回归。

可观测

每次运行产生完整的 state.json + meta.json + 各龙虾输出。任何扩展都不能破坏这个日志链。未来接入监控仪表盘时，数据结构已经就绪。

🧬自迭代引擎

龙虾军团的终极目标：系统能自动发现质量短板、自动生成改进方案、自动验证效果、自动晋升最优版本。人类只需要做两件事——提供标注数据和最终审批。

自迭代的三个陷阱和防护机制

陷阱一：过拟合到测试集

对 10 个测试 Idea 越来越好，但遇到新 Idea 就崩。这是自迭代系统最常见的死因。

防护：

① 测试集分为训练集（7个）和留出集（3个）。变异/竞赛只用训练集，晋升前必须在留出集上通过。

② 每月更换 2 个测试 Idea。用新 Idea 的表现作为"泛化能力"指标。

③ 记录每个 Idea 被用于迭代的次数。超过 10 次的 Idea 降权。

陷阱二：变异震荡

第 N 轮加了一条指令，分数提高；第 N+1 轮诊断发现新问题，加了另一条指令，结果第一条的效果被抵消了。反复震荡。

防护：

① 每只龙虾的追加指令不超过 3 条。超过时必须合并或替换，不允许无限堆叠。

② 每条追加指令标记生效日期。超过 5 轮未被验证有效的指令自动标记为"待清理"。

③ 每 10 轮做一次"Prompt 瘦身"：用 Sonnet 重写整个 System Prompt，合并冗余指令。

陷阱三：评估偏见放大

L2 LLM 评估层有偏见（偏好长文本、偏好格式整齐的内容）。如果迭代完全依赖 L2，系统会朝着"看起来专业但内容空洞"的方向优化。

防护：

① 自动晋升只允许基于 L1（规则层）的判断。L2 分数只作为参考信号。

② 每 5 轮自动晋升后，必须有一次 PM 的 L3 人工审核。如果人工分数与自动分数偏差 > 2 分，触发"评估校准"流程。

③ 每季度用最新的人工标注数据重新校准 L2 的评估 Prompt。

════════ Phase 2: 独立压缩 + 合并 ════════ @listen(run_lobster_1) def compress_l1(self): s1 = compress_with_fallback(self.state.L1.content) self.state.summary_l1 = s1 return s1 @listen(run_lobster_2) def compress_l2(self): s2 = compress_with_fallback(self.state.L2.content) self.state.summary_l2 = s2 return s2 @listen(and_(compress_l1, compress_l2)) # ← 条件路由：检查点 B def gate_check(self): """语义一致性校验：龙虾三输出 vs 原始 Idea""" score = check_semantic_consistency( self.state.anchor, self.state.L3.content ) self.state.gate_score = score if score >= 5: return "passed" # → run_lobster_4 elif self.state.gate_retries < 1: return "retry" # → retry_lobster_3 else: return "abort" # → assemble_partial @listen("retry") def retry_lobster_3(self): """换 temperature 0.5 重跑龙虾三""" self.state.gate_retries += 1 agent = create_lobster_3(self.state.anchor, temperature=0.5) ... # 同 run_lobster_3 逻辑 return result # → 再次触发 gate_check @router @listen("abort") def assemble_partial(self): """连续两次 gate 失败 → 返回部分报告""" self.state.phase = "partial" self.state.persist() return build_partial_report(self.state) # ════════ Phase 4-5: 龙虾四五串行 ════════ @listen("passed") def run_lobster_4(self): agent = create_lobster_4(self.state.anchor) task = Task( description=build_l4_prompt( self.state.anchor, self.state.L3.content, # 龙虾三全文 self.state.combined_summary # 市场摘要 ), expected_output="3 套技术方案 + 成本估算", agent=agent, ) crew = Crew(agents=[agent], tasks=[task]) result = crew.kickoff() self.state.L4 = parse_output(result) self.state.persist() return result @listen(run_lobster_4) def run_lobster_5(self): agent = create_lobster_5(self.state.anchor) task = Task( description=build_l5_prompt( self.state.anchor, self.state.L3.content, # 龙虾三全文 self.state.L4.content, # 龙虾四全文 self.state.combined_summary # 全部摘要 ), expected_output="风险分析 + 投资人视角", agent=agent, ) crew = Crew(agents=[agent], tasks=[task]) result = crew.kickoff() self.state.L5 = parse_output(result) self.state.persist() return result # 关键设计决策

为什么用 Flow 而不是单个 Crew？

单个 Crew 只支持 sequential 或 hierarchical 两种执行模式。龙虾军团需要的是"并行→压缩→串行→条件路由→串行"的混合拓扑，只有 Flow 能表达。每只龙虾包在一个单 Agent 的 Crew 中，由 Flow 编排 Crew 之间的执行顺序和数据传递。

为什么每只龙虾是单独的 Crew？

CrewAI 的 context=[task] 只在同一 Crew 内生效。龙虾之间的数据传递（锚点、摘要、全文）需要精确控制注入哪些内容，用 Flow 的 state 手动传递比 Crew 内部的自动 context 更可控。每只龙虾 = 一个 Crew = 一次独立的 LLM 调用，token 预算可精确管控。

压缩层为什么不用 Agent？

压缩步骤是 Haiku 的单次 API 调用 + JSON Schema 校验。不需要 Agent 的工具调用、推理循环等功能。在 Flow 中用普通 Python 方法直接调 Anthropic SDK 更高效，避免 CrewAI Agent 的 max_iter 循环开销。

web_search 怎么接入？

CrewAI 的 Anthropic Provider 原生支持 Claude 的 tool_use。在 Agent 定义中不需要额外配置——Sonnet 4.6 自带 web_search 能力。只需在 Task description 中指示"使用搜索验证数据"，Claude 会自行调用。搜索结果的截断和容错在 Flow 的 compression 步骤中处理。

五、优化手册（解决常见问题）

遇到问题时，按以下手册排查。每个症状都有明确的诊断路径和修复方案。

🔥
症状：龙虾输出跑偏，不回答正确问题
最常见问题，通常在调试初期出现
排查顺序 ① 检查锚点注入：打印实际发送给 API 的 messages，确认 product_anchor 在 System Prompt 开头。
② 检查 Few-shot：Few-shot 示例是否和目标行业匹配？如果 Idea 是硬件产品但 Few-shot 全是 SaaS 案例，龙虾会朝 SaaS 方向跑。
③ 检查上游数据：龙虾三以后的龙虾，检查注入的摘要/全文是否正确。用 console.log 打印实际注入内容。
④ 降低 temperature：如果以上都正确，尝试 temperature 从 0.7 降到 0.3。低温度减少创造性但提高忠实度。
📉
症状：输出质量不稳定，同一 Idea 跑三次差异大
方差问题，影响系统可靠性
